# LunaPlate 实用性架构方案（迭代总纲）

> 本文件是产品与技术的单一事实来源。每次迭代前先读本文件，完成一项就把对应任务的复选框勾上并在「迭代日志」追加一行。不确定优先级时，永远按 Phase 顺序从上往下做第一个未完成项。

---

## 1. 产品定位

给女性用户的「经期感知型日常护理」App：用户记录经期与症状，App 基于周期阶段（月经期 / 卵泡期 / 排卵期 / 黄体期）+ 当日症状，给出饮食、运动、护理（睡眠 / 保暖 / 护肤 / 情绪）建议，并提前提醒下次经期。

**定位边界（不可违反）**：健康生活建议工具，不做医疗诊断。所有建议附非诊断声明；出现红旗症状时只做「建议就医」提示，不给处置方案。

## 2. 现状与差距

已有：阶段估算（末次经期 + 平均周期长度）、手动阶段覆盖、症状勾选、TheMealDB 餐食推荐与打分、本地运动库推荐、今日计划 / 饮水打卡、购物清单、三语 i18n（en/zh/ru）、PWA 离线壳、Render 部署。

核心差距（为什么现在「不实用」）：

1. **没有真正的经期记录**——只有一个「末次经期日期」，用户无法记录每次经期的开始 / 结束，预测不会随使用变准。这是经期 App 的立身之本。
2. **症状打卡不留历史**——勾了就丢，无法回看「我每次黄体期都头痛」这类规律。
3. **数据只在 localStorage**——换手机 / 清缓存即全丢，无导出导入。
4. **没有提醒**——经期 App 最高频的实用需求是「提前提醒我」。
5. **护理建议单薄**——只有餐食和运动，缺睡眠 / 保暖 / 护肤 / 情绪等护理维度，且建议不解释「为什么」。
6. **强依赖 TheMealDB**——外网接口挂了今日推荐就空白，无离线回退。
7. **零测试**——打分、阶段计算等纯逻辑没有任何回归保障。

## 3. 架构原则（约束 codex 的每次改动）

- **保持零构建、零第三方依赖**：前端原生 JS，后端 Python 标准库。引入任何依赖 / 构建工具需先在本文件「决策记录」里写明理由。
- **Local-first**：用户数据（周期、症状、打卡）以浏览器本地存储为权威副本；服务端保持无状态（Render 免费实例磁盘是临时的，会随重新部署清空，**禁止在服务端落盘用户数据**）。云同步属 Phase 4，且必须接外部数据库才能做。
- **同域单服务**：静态文件 + `/api/*` 同一个 `server.py` 提供，保住 PWA 同源缓存模型。
- **数据 schema 版本化**：所有 localStorage 结构带 `schemaVersion` 字段，升级时写迁移函数，禁止破坏老数据。
- **内容与代码分离**：推荐规则、护理内容全部放 `data/*.json`，代码只做引擎；新增建议内容不改 JS/Python 逻辑。
- **i18n 完整性**：任何新增用户可见文案必须同时提供 en/zh/ru 三份，放 `i18n.js`；缺翻译视为任务未完成。
- **小步提交**：每个任务一个 commit，commit message 用英文祈使句并引用任务编号（如 `P1-2: store cycle history`）。

## 4. 数据模型（localStorage，目标形态）

```jsonc
// key: lunaplate.cycles.v1
{
  "schemaVersion": 1,
  "cycles": [
    // 按 startDate 升序；endDate 可为 null（当前经期进行中）
    { "startDate": "2026-06-05", "endDate": "2026-06-10" }
  ],
  "settings": { "avgCycleLength": 28, "avgPeriodLength": 5, "manualOverride": null }
}

// key: lunaplate.dailyLog.v1
{
  "schemaVersion": 1,
  "days": {
    "2026-07-03": {
      "symptoms": ["cramps", "fatigue"],
      "flow": "medium",          // none|spotting|light|medium|heavy
      "mood": "low",             // great|ok|low|irritable|anxious
      "water": 1200,             // ml
      "notes": "",
      "planChecks": ["warm-compress"]
    }
  }
}
```

预测算法（P1-3）：取最近 6 个完整周期算长度均值与标准差；不足 2 个周期时回退到用户设置的平均值。标准差 > 7 天标记「周期不规律」，预测展示为日期区间而非单日。

## 5. 路线图

### Phase 1 — 核心记录与预测（让它成为「真的经期 App」）

- [x] **P1-1 周期数据层**：新建 `cycle-store.js`（或 app.js 内独立模块），实现上面 `lunaplate.cycles.v1` 的读写、迁移（把现有「末次经期日期」迁成第一条 cycle 记录）、增删改。
- [ ] **P1-2 经期记录 UI**：日历视图上支持「标记经期开始 / 结束」，历史经期在日历上着色；支持补录和修改过往经期。
- [ ] **P1-3 基于历史的预测**：按上面算法预测下次经期、排卵日、易孕窗口；日历上区分「实际记录」与「预测」（实心 vs 虚框）；阶段估算改为基于最近一条实际周期而非手填日期。
- [ ] **P1-4 每日打卡历史化**：症状 / 流量 / 心情 / 饮水按天写入 `lunaplate.dailyLog.v1`，可回看与修改任意一天；今日页默认打开当天。
- [ ] **P1-5 数据导出 / 导入**：设置页提供导出全部数据为 JSON 文件、从 JSON 恢复（含 schema 校验与版本迁移）。这是换机 / 备份的兜底，必须先于一切云功能存在。

**验收**：新用户连续模拟记录 3 个周期后，预测日期随记录更新；清空 localStorage 后用导出文件可完整恢复；老用户数据自动迁移不丢。

### Phase 2 — 护理推荐引擎（从「推荐菜」到「推荐护理方案」）

- [ ] **P2-1 护理内容库**：新建 `data/care-rules.json`，结构为 `阶段 × 症状 → 建议卡`，每条建议含：类别（nutrition/movement/sleep/warmth/skincare/mind）、标题、做法、**一句话原因**（如「黄体期孕酮上升易水肿，建议减盐」）、i18n key。首批覆盖 4 阶段 × 8 常见症状（痛经、腹胀、头痛、疲劳、失眠、情绪波动、痤疮、乳房胀痛），每格至少 2 条建议。
- [ ] **P2-2 推荐引擎与今日护理页**：`/api/care-plan` 按阶段 + 当日打卡症状返回排序后的建议卡；今日页顶部展示 3–5 张「今日护理」卡，可标记完成（写入 planChecks）。
- [ ] **P2-3 红旗提示**：`data/red-flags.json` 定义规则（如「单周期流量 heavy ≥ 7 天」「打卡剧烈疼痛」「周期 < 21 天或 > 45 天连续 2 次」），命中时展示温和的就医建议横幅，文案经三语翻译，永远不给诊断结论。
- [ ] **P2-4 餐食推荐可解释**：现有餐卡增加「为什么推荐」一行（复用打分理由：阶段营养标签 / 症状匹配 / 忌口过滤）。

**验收**：勾选「痛经 + 月经期」能看到热敷 / 镁 / 轻柔运动等分类建议且各有原因；触发红旗规则出现就医提示；所有新文案三语齐全。

### Phase 3 — 提醒与洞察（让用户每天回来）

- [ ] **P3-1 本地提醒**：基于 Notification API + Service Worker 的本地提醒：经期前 2 天、预测当天、排卵日；iOS PWA 通知受限，需在设置页说明并提供「添加到系统日历（.ics 下载）」作为替代。
- [ ] **P3-2 周期洞察**：Insights 页做成真实数据图表（纯 SVG，不引库）：周期长度趋势、经期长度趋势、症状 × 阶段热力表（「你的头痛 80% 出现在黄体期」）。
- [ ] **P3-3 不规律提醒**：检测到周期不规律（P1-3 的标准差规则）时，洞察页给出说明性提示与就医建议阈值（连续异常再提示，避免焦虑）。

**验收**：安卓 Chrome 安装 PWA 后能收到经期前提醒；有 3+ 周期数据时洞察图表正确渲染；无数据时有引导性空状态。

### Phase 4 — 可靠性与可持续

- [ ] **P4-1 离线餐食回退**：精选 40–60 道菜谱快照存 `data/fallback-meals.json`（按阶段 × 餐型覆盖），TheMealDB 超时 / 失败时用本地库走同一套打分；今日推荐永不空白。
- [ ] **P4-2 后端测试**：pytest 覆盖 `score_meal`、`infer_*`、阶段计算、care-plan 引擎、红旗规则；加 `tests/` 目录与 `python -m pytest` 说明。pytest 是允许引入的唯一开发依赖。
- [ ] **P4-3 前端逻辑测试**：把预测 / 迁移等纯函数抽到可独立加载的模块，用 Node 原生 `node --test` 跑（不引测试框架）。
- [ ] **P4-4 隐私声明**：设置页加隐私说明（数据仅存本机、服务端不留存）；这是经期数据的敏感性要求，也是上架 / 传播的前提。
- [ ] **P4-5（可选，需决策）账户与云同步**：需外部托管数据库（如 Supabase/Turso 免费档）+ 端侧加密。做之前必须在「决策记录」写明方案并确认放弃「零依赖」原则的范围。

**验收**：断网状态下今日页仍有完整推荐；`python -m pytest` 与 `node --test` 全绿；CI 可后续接 GitHub Actions。

## 6. 每次迭代的工作流（codex 执行规范）

1. 读本文件，找到第一个未勾选任务。
2. 实现 → 本地跑 `python server.py` 手动验证核心路径（记录 → 预测 → 推荐）→ 有测试则跑测试。
3. 勾选任务、追加迭代日志、提交 commit。
4. 一次迭代只做一个编号任务；发现前置缺陷可先修并记为 `FIX-n`。
5. 涉及原则冲突（加依赖、服务端存数据等）：停下，在「决策记录」写明再动手。
